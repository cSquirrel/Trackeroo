# Trackeroo (Tauri desktop app)

> This file covers conventions specific to the implementation — not the product overview, quickstart, or repo layout, which live in `README.md`. All paths below are relative to `v2/` — `cd` into `v2/` before running any commands here.

See `README.md` for the user-facing picker flow (New / Open / Recent), what a "project" is (a folder with a `.trackeroo/` state subfolder), and where data lives. This file covers the *why* and the gotchas.

## Stack

- **Shell**: Tauri v2 (Rust), `src-tauri/src/lib.rs`. Shows the window immediately on launch (`visible: true` in `tauri.conf.json`) with **no backend running** — the webview renders the project picker, which needs no backend. A per-project backend is spawned only when the user picks a project, via the `open_project` command. On exit the backend is killed — both GUI quit (`RunEvent::Exit`/`RunEvent::ExitRequested`) and POSIX signals (SIGINT/SIGTERM/SIGHUP via a `ctrlc` handler), since a bare `kill` of the app process wouldn't otherwise reap the backend child.
- **Backend**: FastAPI + SQLAlchemy + SQLite. It reads `DATABASE_URL` and `TRACKEROO_PORT` from the environment, which is all the shell needs to point one backend at one project's DB on one port. Dev mode spawns it from `backend/.venv/bin/uvicorn` directly (`cfg(debug_assertions)` path in `lib.rs`); release mode spawns a PyInstaller-bundled binary next to the app executable (`cfg(not(debug_assertions))` path). `backend/run_sidecar.py` is the PyInstaller entry point (kept separate from `app/` so PyInstaller has a concrete script target).
- **Frontend**: Svelte + TypeScript, adapted for the Tauri webview. `frontend/src/App.svelte` is a gate: it renders `ProjectPicker.svelte` until a project is opened, then `BoardApp.svelte` (the former single-project root). The API base URL in `frontend/src/lib/api.ts` is a **runtime** value set via `setApiBaseUrl()` — see the init sequence below. Vite dev server is pinned to port `1420` (`strictPort: true`) since Tauri's `devUrl` in `tauri.conf.json` expects a fixed port.
- **MCP server**: `mcp/server.py`, a thin HTTP client of the backend's REST API — discovers its target by **project folder** instead of a fixed URL (`TRACKEROO_API_URL` would go stale the moment the app restarts on a different port). Set `TRACKEROO_PROJECT_PATH` to a project folder and it reads `<project>/.trackeroo/.env` fresh before every request, self-healing across app restarts with no config changes. `TRACKEROO_API_URL` is still supported as a direct override (takes priority if set) for cases where a fixed URL is actually correct. **Bundled into the app itself** as a third Tauri `externalBin` (`trackeroo-mcp`, alongside `trackeroo-backend`) — same PyInstaller pattern as the backend sidecar, so the installed app is fully self-contained (no dev checkout, no Python/pip/venv needed on the target machine to use MCP). Build it with `--collect-submodules mcp.server`, not the whole `mcp` package — `mcp.cli` is an optional submodule that imports `typer`, which isn't a dependency and isn't needed (only `mcp.server.fastmcp` is used).

## How multi-project works

- **Project state layout.** All of a project's Trackeroo-owned state lives under `<folder>/.trackeroo/` — `trackeroo.db` and, while a backend is running, `port`. `state_dir()`/`db_path_for()` in `lib.rs` are the only places that should ever construct these paths; don't hardcode `.trackeroo/...` elsewhere. **Migration is load-bearing, not optional cleanup**: `migrate_legacy_layout()` runs at the top of `open_project`, and if `.trackeroo/trackeroo.db` doesn't exist yet but a loose `<folder>/trackeroo.db` does (the pre-`.trackeroo/` layout), it moves the old file into the new location before anything else touches it. Skipping or removing this would silently orphan every project folder created before this layout existed — the app would either error "no project found" (Open) or create a second, empty database next to the real one (New/CLI-launch, since `create: true` doesn't check for existing data at the new path). `list_recent_projects` checks both the new and legacy path for exactly this reason — an unmigrated recent entry (not yet reopened) must not silently disappear from the list.
- **Dynamic port, one backend per process.** There is no fixed backend port anymore. `pick_free_port()` in `lib.rs` binds a `TcpListener` to port 0, reads the OS-assigned port, drops the listener, and that port is passed to the spawned backend (`--port` in dev, `TRACKEROO_PORT` in release). This is what lets multiple project windows coexist without colliding. The chosen port is stored in a global (`BACKEND_PORT`) and exposed to the frontend via the `get_backend_port` command.
- **Port discovery for MCP.** `open_project` also writes the chosen port to `<project-folder>/.trackeroo/.env` (`TRACKEROO_PORT=<port>`, KEY=VALUE format so more fields can be added later without changing the format again) once the backend is healthy, and `kill_backend()` removes it — so the file only exists while a backend for that project is actually running. `mcp/server.py` reads and parses it by project path instead of needing a hardcoded port (see "MCP server" above; `_parse_env_file()` is a deliberately minimal line-based parser, not a dependency). `PROJECT_FOLDER` (a new global alongside `BACKEND`/`BACKEND_PORT`) tracks which folder's `.env` to clean up on exit.
- **Frontend runtime-config init sequence (order matters):** the picker calls the `open_project` command → the shell spawns the backend and returns the port → `App.svelte` calls `setApiBaseUrl("http://localhost:<port>/api")` → *only then* does `BoardApp` mount and call `store.loadAll()`. Nothing fetches before the base URL is configured. For a *newly created* project, `App.svelte` also PATCHes the project name (the backend seeds a default "Trackeroo" name; we override it with what the user typed).
- **Separate process per project — no handoff.** "Open project…" in the board top bar calls the `open_new_window` command, which does `std::env::current_exe()` + `Command::spawn()` (direct execution, detached, not waited on). That new OS process starts completely fresh and shows its own picker; the chosen project is **not** passed to it. Direct exec is deliberate: on macOS, `open`-ing the `.app` refocuses an already-running instance instead of launching a new one, whereas direct exec of the binary always yields a new independent process (`LSMultipleInstancesProhibited` is absent, so multiple instances are allowed).
- **Recent-projects storage.** `list_recent_projects` / `record_recent` in `lib.rs` persist a JSON array of `{ "path", "name" }` (most-recent-first, de-duped, capped at 10) to `recent_projects.json` under Tauri's `app_config_dir()` (`~/Library/Application Support/com.trackeroo.desktop/`). `list_recent_projects` filters out entries whose database no longer exists at either the current `<path>/.trackeroo/trackeroo.db` location or the pre-migration `<path>/trackeroo.db` location.
- **CLI launch path.** `compute_launch_target()` in `lib.rs` runs once in `run()` (result cached in `LAUNCH_TARGET`) and interprets the first non-flag CLI arg as a project folder: it must exist and be a directory (canonicalized; flag-like args such as macOS's legacy `-psn_...` are skipped). The frontend calls `get_launch_target` on mount (`App.svelte`) — if a `path` came back it opens that project directly via the *same* `open_project` command the picker uses (`create: true`, so a fresh folder becomes a new project and an existing database just loads), skipping the picker; if an `error` came back (invalid path) it shows the picker with that error inline. This is only for external launches; `open_new_window` spawns a bare instance with no args, so a CLI path is never handed to the "open a different project" button.

## Conventions

- Data model / REST contract is documented in `docs/api-contract.md` and `docs/openapi.json`. The latter is generated, never hand-edited — regenerate it after any route/schema change: `cd backend && .venv/bin/python -m app.export_openapi`, and commit the diff alongside the code change that produced it.
- The backend port is chosen dynamically at spawn time (see "How multi-project works" above) — do not reintroduce a hardcoded port. The DB path likewise comes from the picked project folder's `.trackeroo/` subfolder (see "Project state layout" below), passed as `DATABASE_URL=sqlite:///<abs>/.trackeroo/trackeroo.db`.
- **Cold-start retry is load-bearing, not defensive boilerplate.** `frontend/src/lib/store.svelte.ts`'s initial `loadAll()` retries the first fetch for up to 30s (60 attempts × 500ms). This fixes a real, verified bug: the webview starts fetching before the freshly-spawned sidecar is guaranteed to be listening, and the release build's PyInstaller sidecar has real self-extraction overhead (~9s measured). It applies per project-open now (the backend spawns when a project is picked, not at app launch). Removing or shortening this without understanding why it's there will reintroduce a "Load failed" banner.
- **Backend kill logic must target the whole process group.** The release sidecar binary is a PyInstaller bootloader that forks the real Python server as a child process — killing only the direct child (the bootloader) orphans the server and leaves its port held. `kill_backend()` in `lib.rs` sends `SIGKILL` to the negated PID (the process group) for exactly this reason; the backend is spawned with `process_group(0)` to make itself a group leader. `open_project` also calls `kill_backend()` before spawning, so re-opening within one process never leaks a backend.
- Sidecar binary naming must exactly match Tauri's `<name>-<target-triple>` convention (currently `trackeroo-backend-aarch64-apple-darwin`) — see `bundle.externalBin` in `src-tauri/tauri.conf.json`. The compiled binary and PyInstaller build artifacts are gitignored; rebuild them after a fresh clone or backend change (see `README.md`).

## Testing & running locally

`cd backend && pytest --cov=app --cov-report=term-missing` and `cd frontend && npm run test` for unit tests. **No E2E suite or CI wiring exists for this project yet** — verify changes manually via `npm run dev`, or by rebuilding and launching the `.app` — a cold-start launch, not just `tauri dev`, is the more representative test given the cold-start convention above. See `README.md` for the full dev/build setup including the one-time backend venv and PyInstaller steps.
